## ğŸ”’ RLS Policy Logic

### Profiles Table

```sql
-- SELECT Policy
USING (
  domain_id = (
    SELECT domain_id
    FROM profiles
    WHERE id = auth.uid()
  )
)
-- Translation: "You can only see profiles in YOUR domain"
```

### Posts Table

```sql
-- SELECT Policy
USING (
  domain_id = (
    SELECT domain_id
    FROM profiles
    WHERE id = auth.uid()
  )
)
-- Translation: "You can only see posts in YOUR domain"

-- INSERT Policy
WITH CHECK (
  auth.uid() = author_id AND
  domain_id = (
    SELECT domain_id
    FROM profiles
    WHERE id = auth.uid()
  )
)
-- Translation: "You can only create posts with YOUR domain"
```

### Likes Table

```sql
-- INSERT Policy
WITH CHECK (
  auth.uid() = user_id AND
  EXISTS (
    SELECT 1 FROM posts p
    INNER JOIN profiles pr ON pr.id = auth.uid()
    WHERE p.id = post_id
    AND p.domain_id = pr.domain_id
  )
)
-- Translation: "You can only like posts in YOUR domain"
```

### Connections Table

```sql
-- INSERT Policy
WITH CHECK (
  auth.uid() = follower_id AND
  EXISTS (
    SELECT 1 FROM profiles p1, profiles p2
    WHERE p1.id = follower_id
    AND p2.id = following_id
    AND p1.domain_id = p2.domain_id
  )
)
-- Translation: "You can only follow users in YOUR domain"
```

## ğŸ¯ Key Security Features

1. **Database-Level Enforcement**: Domain gate is enforced by PostgreSQL RLS, not just app logic
2. **Automatic Domain Inheritance**: Posts automatically inherit author's domain via trigger
3. **Unique Constraints**: Prevent duplicate likes and connections
4. **Cascade Deletes**: Clean up related data when users/posts are deleted
5. **Check Constraints**: Ensure only valid domains are used
6. **Foreign Keys**: Maintain referential integrity across tables
